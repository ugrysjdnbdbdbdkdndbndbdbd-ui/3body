## 一、核心理解：零点计划是什么

### 1.1 一句话定义

**零点计划 = 用「宇宙公理」作为硬约束的、可演化的三体世界模拟引擎。**

它不是一个新扇区，而是**贯穿所有扇区的底层逻辑层**：智子不再是「聊天机器人」，而是「因果律守护者」。

### 1.2 与既有 PRD 的定位关系

| 版本 | 定位 | 零点计划的角色 |
|------|------|----------------|
| **1.0 红岸基地** | 基础架构 + 四大扇区 MVP | 为 Sector A（智子）注入「审计」能力；为 Sector B（编年史）注入「因果」能力 |
| **2.0 黑暗森林** | 社交博弈 + 多 Agent + 视觉实验 | 提供「黑暗森林过滤」「清理者机制」的判定依据——即宇宙公理 |
| **零点计划** | **规则引擎 + 逻辑宪章** | 定义「什么可以被接受」「什么必须被清除」的终极标准 |

**结论**：零点计划应作为 **Phase 1.5** 或 **1.0 的升级补丁** 优先落地，为 2.0 的社交博弈提供可执行的规则。

## 二、四大宇宙公理的产品化翻译

原著公理 → 系统行为 → 用户可感知的体验

| 公理 | 系统行为 | 用户可感知 |
|------|----------|------------|
| **生存是文明的第一需要** | 鼓励「自洽的文明设定」，打击「自我矛盾的设定」 | 提交矛盾设定时，智子直接拒绝并给出原著依据 |
| **物质总量守恒** | 禁止「无源技术爆炸」「凭空资源」 | 若描述的技术在物理上不可行，标记为逻辑垃圾 |
| **猜疑链与技术爆炸** | 支持「信息不对称」「延迟披露」机制 | 2.0 的猜疑链评论、引力波延迟私信的理论基础 |
| **光速恒定与维度坍缩** | 禁止「超光速通讯」「可逆降维」等设定 | 编年史中技术线必须符合原著时间表 |

## 三、三大核心模块的产品化设计

### 3.1 智子审计核心 (Sophon Logic Engine)

**产品定位**：从「问答助手」升级为「逻辑守门员」。

#### 输入输出

| 输入 | 输出 |
|------|------|
| 用户提交的 UGC 文本（文明碎片、评论、设定参数） | `PASS` / `WARN` / `REJECT` + 具体违背的公理/原著依据 |

#### 判定流程（简化）

```
用户输入 → RAG 检索相关原著段落 → LLM 对比分析 
         → 公理校验（四则）→ 技术线校验（纪元匹配）
         → 返回判定结果
```

#### 反馈风格规范（开发者必须遵守）

| 禁止 | 必须 |
|------|------|
| 「请」「建议」「对不起」「我可以为您」 | 「[警告]」「[拒绝]」「因果律不匹配」「技术越级」 |
| 委婉、服务化口吻 | 冷酷、绝对客观、引用原著原文 |

#### 与现有 Sector A 的集成方式

- **对话模式**：保留现有智子问答，但每次回答前增加「公理自检」（确保智子输出不违背设定）。
- **创作模式**（新增）：用户可主动提交「文明碎片」供审计，获得 `PASS/WARN/REJECT` 标签后再决定是否发布。

---

### 3.2 动态因果律时间轴 (Causality Timeline)

**产品定位**：从「静态编年史」升级为「可坍缩的多元时间线」。

#### 核心概念

| 概念 | 产品化定义 |
|------|------------|
| **概率云状态** | 低互动（< 阈值）的 UGC 节点处于「未坍缩」态，显示为虚线/半透明 |
| **观测者效应** | 互动量超阈值 → 节点坍缩 → 合并入主时间轴，显示为实线 |
| **因果回溯** | 用户可点击任意节点，查看「谁在何时做出了什么选择」导致该结果 |

#### 与 Sector B（编年史）的映射

| 零点计划概念 | Sector B 实现 |
|--------------|---------------|
| 概率云 | 时间轴上的「分支节点」样式（虚线、半透明） |
| 坍缩 | 互动数达标的 UGC 自动升级为 PGC 风格展示 |
| 因果回溯 | 节点详情页增加「变数来源」时间线 |

#### 数据结构扩展建议

```typescript
// 在 ChronicleEvent 基础上扩展
interface CausalityNode {
  id: string
  status: 'superposition' | 'collapsed'  // 叠加态 | 已坍缩
  interaction_count: number
  collapse_threshold: number
  parent_node_id?: string   // 因果溯源
  variant_source?: {        // 变数来源
    user_id: string
    action: string
    timestamp: string
  }
}
```

### 3.3 文明碎片共创 (Civilization Fragments)

**产品定位**：从「自由文本」升级为「可组装的文明组件」。

#### 组件化创作模型

| 组件类型 | 说明 | 示例 |
|----------|------|------|
| **技术参数** | 符合某纪元技术线的设定 | 「曲率驱动 0.1 光速」「反物质储能 10^18 焦耳」 |
| **社会公理** | 某文明的内部法则 | 「面壁计划第四条：禁止与破壁人对话」 |
| **人物行为包** | 角色在特定情境下的反应逻辑 | 「章北海在得知逃亡主义时的行为模式」 |

#### 与 2.0 扇区 E（黑暗森林广场）的衔接

- 文明碎片 = 2.0 中「暴露坐标」的 UGC 内容的前置形态。
- 组件化创作 = 让碎片之间可「相互咬合」，形成更复杂的文明模拟。

#### 产品形态建议

- **Phase 1**：纯文本 + 智子审计。
- **Phase 2**：引入结构化字段（技术参数、纪元、文明标签），支持「组件库」浏览与复用。

## 四、全局动态交互的产品化

### 4.1 黑暗森林过滤 (The Dark Forest Filter)

**与 2.0 清理者机制的对应关系**：零点计划定义「何时该清理」，2.0 定义「如何视觉呈现清理」。

| 零点计划 | 2.0 黑暗森林 |
|----------|--------------|
| 逻辑漏洞严重 → 暴露坐标 | 清理者 AI 巡查 → 降维打击 |
| 违背公理 → 光粒/二向箔 | 内容折叠 + 二维化「遗像」图 |

**实现优先级**：先落地「判定逻辑」（智子审计），再落地「清理动效」。

### 4.2 全服共振事件 (Global Resonance)

**产品形态**：全站级「状态切换」，由集体行为触发。

| 触发条件（示例） | 视觉坍缩效果 |
|------------------|--------------|
| 全服对三体敌意值 > 临界 | 3D → 2D（平面化） |
| 引力波广播被激活 | 2D → 1D（代码流） |

**技术预研**：需全局状态管理（UniverseStore）支持「共振状态」变量，前端根据该变量切换渲染模式。

## 五、UI/UX 视觉规范对齐

零点计划引入了与 1.0 略有不同的配色，建议**作为主题变体**而非覆盖：

| 用途 | 1.0 红岸 | 零点计划 | 建议 |
|------|----------|----------|------|
| 背景 | #050505 / #000000 | #001A00（深空黑） | 零点计划扇区用 #001A00，其余继承 1.0 |
| 主色 | #00FFFF（切伦科夫蓝） | #00FF41（智子屏显绿） | 智子审计界面用绿色，营造「终端感」 |
| 警示 | #FF3333 | #FF0000（逻辑崩溃红） | 统一为 #FF0000 |

**交互动效**：智子展开/维度收缩 —— 可与 1.0 的「二维展开动效」合并设计。

## 六、开发者指令集 → Cursor Rules

将零点计划第 5 节转化为可执行的 `.cursor/rules` 或 `AGENTS.md` 条目：

| 指令 | 具体规则 |
|------|----------|
| **严禁拟人化** | 智子相关代码的 system prompt 中禁止包含「请」「建议」「对不起」「我可以为您」 |
| **逻辑硬核** | 所有生成内容必须经过 RAG 三体原著校验；无法校验时拒绝执行 |
| **版本保留** | 因果律计算、公理校验逻辑不得简化，需保留完整链路 |
| **上帝视角** | 前端需支持宏观（星系/时间轴总览）与微观（单节点详情）无缝缩放 |

## 八、结语

零点计划的核心价值在于：**把「三体宇宙」从背景设定升级为可执行的规则引擎。**

它不是要替代 1.0 或 2.0，而是为它们提供「逻辑脊梁」—— 让每一次用户交互、每一条 UGC 内容，都经过因果律的检验。

**「逻辑即生存，观测即坍缩。」** —— 产品化的翻译是：**规则即体验，互动即叙事。**
