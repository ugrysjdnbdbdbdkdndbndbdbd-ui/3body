# 3Body X 本地开发规则

本文档与 `.cursor/rules/3body-agent.mdc` 一致，作为团队与 Agent 的开发原则参考。

## 自适应与布局（Layout & Responsiveness）

### 下方展示与屏幕宽度自适应

**原则**：主内容区（如三体体验沙盒各 Tab 下的面板、列表、图表、表单）**必须与视口宽度自适应**。

- **禁止**：固定像素宽度导致大屏两侧留白过多，或小屏横向溢出、需左右滚动。
- **要求**：
  - 使用 `max-width` 配合 `width: 100%`，使内容在宽屏居中且随容器收缩。
  - 栅格与列表使用 `grid-template-columns: repeat(auto-fill, minmax(...))` 或 `clamp()`、`minmax()`，随视口变化列数/宽度。
  - 必要时使用媒体查询（如 `@media (max-width: 768px)`）调整内边距、字号、单列布局。
  - 图表、SVG 使用 `viewBox` + `width: 100%` 或 `max-width: 100%` 保持比例与自适应。

**适用**：三体体验沙盒（SectorZeroV2）、黑暗森林沙盒、编年史、画廊等所有「下方」主展示区域。

## 体验沙盒 · 游戏化设计原则（Game Design Principles）

**定位**：三体体验沙盒本质是一款**游戏**。所有该模块的设计与开发须以**资深游戏策划**视角，围绕参与性、数值体系、可玩性展开，并作为该模块的强制开发原则。

### 1. 参与性（Participation / 门槛）

- **零认知即可开玩**：新用户进入任一子模块（面壁者 / 黑暗森林 / 云天明 / 思想钢印）后，应在**不阅读长文案**的前提下，通过**预设选项、一键操作、明确按钮**在 10 秒内完成第一次有效操作（如「选择一条指令并提交」「选择一则童话并解密」「点击运行」）。
- **选项优先于输入**：核心玩法必须提供**可选的预设路径**（快捷指令、示例机密、拆解词、预设参数等），输入框作为进阶/自定义补充，而非必经步骤。
- **单次操作即可反馈**：每一次主要操作（提交指令、推进纪元、执行解密、注入钢印等）都应有**即时、可见的数值或状态变化**（进度条、阶段名、列表新增、曲线变化等），避免「点了没反应」或需要多步才看到结果。

**验收**：每个子模块至少支持「仅通过点击/选择完成一局完整循环」，且首屏可见「本局目标」与「关键数值」。

### 2. 数值体系（Numbers & Progression）

- **关键数值显性化**：每个子模块必须有**少量、清晰的核心数值**常驻可见（如面壁者：疑心值 / 90、通过条数；黑暗森林：纪元、存活数、墓志铭数；云天明：解密进度 %、执剑人候补；思想钢印：感染指数、强度）。数值命名与单位统一，避免同义多词（如「进度」与「百分比」二选一并统一）。
- **数值有含义**：每个核心数值都应对应**明确的成败或阶段**（如疑心值 90 = 破壁；解密 80% = 执剑人候补；存活数 0 = 本局结束）。重要阈值在 UI 上可标出（进度条分阶、颜色变化、阶段标签）。
- **可选的元进度**：在不过度复杂化的前提下，可引入**跨局/轻量成就**（如「面壁 N 条未破壁」「解密 M 则童话达 80%」「单局存活至纪元 X」），用于强化目标感与复玩动机；实现方式可为本地存储或仅展示当局/当会话统计。

**验收**：用户能在 3 秒内回答「当前这局的关键数字是什么、赢了/输了看哪个数」。

### 3. 可玩性（Playability / 好玩）

- **目标一句话说清**：每个子模块首屏或玩法说明中必须有**一句「本局目标」**（如「在破壁前尽量多提交通过指令」「让文明存活更久或观察墓志铭」「将童话解密到 80%」），与关键数值一一对应。
- **再玩一局零摩擦**：提供**显眼的「重置 / 再玩一局 / 新局」**入口，一键清空本局状态并回到可操作起点，无需刷新页面或重新接受条款。
- **策略与反馈**：在保留三体叙事的前提下，设计**可感知的策略差异**（如低/中/高风险指令、文明数/纪元步长、不同机密与拆解词组合），并通过数值与文案给予**即时反馈**（通过/驳回、存活曲线、命中高亮），让「下次我换一种玩法」有据可依。
- **节奏与心流**：避免长时间无反馈的等待；耗时操作需有 loading 状态或阶段性提示。单局时长建议可控制在数分钟内完成一个完整循环（一轮面壁/一局沙盒/一则童话解密）。

**验收**：用户能在无说明文档的情况下，完成「理解目标 → 做决策 → 看到结果 → 再玩一局」的闭环，并愿意尝试不同策略。

### 实施与自检

- 新增或改版体验沙盒子模块时，必须对照上述三条（参与性、数值体系、可玩性）做自检；PR 或迭代说明中建议注明对应原则的落实情况。
- 与「下方展示与屏幕宽度自适应」不冲突：游戏化原则约束**内容与交互设计**，自适应约束**布局与视口**，二者同时满足。
